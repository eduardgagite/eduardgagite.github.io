Когда проект растёт, он часто разбивается на несколько модулей: основное приложение, внутренние библиотеки, CLI-утилиты. Каждый модуль — отдельный **go.mod**. Проблема: как разрабатывать их одновременно, не публикуя каждое изменение?

До Go 1.18 для этого использовали директиву **replace** в go.mod, указывая локальный путь к модулю. Это работало, но require было не забыть убрать replace перед коммитом. С Go 1.18 появились **Workspaces** — чистое решение для мультимодульной разработки.

## Проблема: локальная разработка нескольких модулей

Допустим, у вас два модуля: приложение и библиотека.

```text
project/
├── app/
│   ├── go.mod      — module github.com/user/app
│   └── main.go
└── lib/
    ├── go.mod      — module github.com/user/lib
    └── calc.go
```

Приложение импортирует библиотеку. Но **go build** в app/ скачает опубликованную версию lib, а не вашу локальную. Раньше приходилось добавлять в app/go.mod:

```go
replace github.com/user/lib => ../lib
```

И убирать эту строку перед коммитом. Workspaces убирают эту необходимость.

## Создание workspace

```bash
cd project
go work init ./app ./lib
```

Появится файл **go.work** в корне:

```go
go 1.24

use (
    ./app
    ./lib
)
```

Теперь **go build**, **go test** и другие команды в любом месте внутри project/ автоматически используют локальные версии модулей.

## Файл go.work

**go.work** — декларация workspace. Он содержит:
- **go** — минимальная версия Go.
- **use** — список директорий с модулями.
- **replace** (опционально) — замены, аналогичные replace в go.mod, но на уровне workspace.

```go
go 1.24

use (
    ./app
    ./lib
    ./tools/migrate
)

replace github.com/broken/dep => github.com/fixed/dep v1.2.3
```

## Основные команды

### Инициализация

```bash
go work init ./module1 ./module2
```

Создаёт **go.work** с перечисленными модулями.

### Добавление модуля

```bash
go work use ./new-module
```

Добавляет модуль в существующий **go.work**.

### Добавление всех модулей рекурсивно

```bash
go work use -r .
```

Найдёт все **go.mod** в поддиректориях и добавит их в workspace.

### Синхронизация

```bash
go work sync
```

Обновляет **go.sum** файлы всех модулей в workspace, чтобы зависимости были согласованы.

## Типичные сценарии

### Монорепозиторий с несколькими сервисами

```text
monorepo/
├── go.work
├── services/
│   ├── api/
│   │   ├── go.mod
│   │   └── main.go
│   └── worker/
│       ├── go.mod
│       └── main.go
└── pkg/
    └── shared/
        ├── go.mod
        └── utils.go
```

```bash
go work init ./services/api ./services/worker ./pkg/shared
```

Все сервисы сразу видят локальную версию shared-пакета.

### Разработка форка библиотеки

Вы нашли баг в сторонней библиотеке, сделали форк и хотите протестировать исправление локально:

```bash
git clone https://github.com/user/some-lib ../some-lib-fork
go work use ../some-lib-fork
```

Ваш проект будет использовать локальный форк. Когда PR примут — удалите строку из go.work.

## Важные правила

1. **Не коммитьте go.work в репозиторий** (если это не монорепозиторий). Добавьте в .gitignore. Файл go.work — инструмент локальной разработки.

2. **В монорепозитории go.work коммитится**. Это единая точка входа для всех модулей.

3. **CI/CD использует go.work**. Если go.work есть в корне, команды go автоматически его подхватывают. Чтобы отключить: **GOWORK=off go build ./...**.

4. **go.work не заменяет go.mod**. Каждый модуль остаётся самостоятельным. Workspace — это только локальная "линза", через которую Go видит модули.

## Отключение workspace

Если нужно временно собрать модуль без workspace (как в CI):

```bash
GOWORK=off go build ./...
```

Или удалите **go.work**.

## Итого

**go work** — встроенный механизм для одновременной разработки нескольких модулей. Заменяет костыли с **replace** в go.mod. Используйте для монорепозиториев, локальной разработки библиотек и тестирования форков. Не забывайте: **go.work** — для разработки, **go.mod** — для публикации.
