Контейнер запущен, но сайт не открывается. Или база данных падает с ошибкой. Как понять, что происходит внутри "черного ящика"?
У Docker есть три главных инструмента для диагностики.

## 1. Логи (docker logs)

В классических серверах логи пишутся в файлы (**/var/log/nginx/error.log**).
В Docker принято писать логи в **stdout** (стандартный вывод). Всё, что приложение пишет в консоль, Docker сохраняет.

Посмотреть логи контейнера:
```
docker logs my-web
```

Следить за логами в реальном времени (как **tail -f**):
```
docker logs -f my-web
```

Показать последние 100 строк:
```
docker logs --tail 100 my-web
```

## 2. Инспекция (docker inspect)

Эта команда показывает **всю подноготную** контейнера в формате JSON: IP-адрес, пути к файлам, переменные окружения, открытые порты и многое другое.

```
docker inspect my-web
```

Вывод будет огромным. Чтобы найти что-то конкретное (например, IP-адрес), можно использовать **grep**:
```
docker inspect my-web | grep IPAddress
```

## 3. Вход внутрь (docker exec)

Иногда нужно зайти внутрь работающего контейнера, чтобы проверить, создался ли файл конфигурации или доступна ли база данных по сети.

Для этого мы запускаем **новый процесс** (обычно оболочку **bash** или **sh**) внутри уже работающего контейнера.

```
docker exec -it my-web bash
```

Если в контейнере нет bash (например, в образах Alpine), пробуйте **sh**:
```
docker exec -it my-web sh
```

Внутри вы можете использовать обычные команды Linux: **ls**, **cat**, **top**, **ping**.
Чтобы выйти, нажмите **Ctrl+D** или наберите **exit**. Сам контейнер при этом **не остановится**, закроется только ваша сессия терминала.

## Итого

- Что-то сломалось? Сначала смотрите **docker logs**.
- Нужно узнать IP или настройки? Смотрите **docker inspect**.
- Нужно "пощупать" файлы руками? Заходите через **docker exec -it ... bash**.
